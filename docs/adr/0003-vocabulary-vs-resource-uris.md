# ADR-0003: Vocabulary URIs use `APP_ORIGIN`

**Date:** 2026-05-11
**Status:** Accepted
**Discussion:** https://github.com/lucas42/lucos_arachne/issues/482

## Context

A lucos service that emits RDF into arachne is typically emitting two distinct kinds of URI:

- **Resource URIs** identify data — `https://media-metadata.l42.eu/tracks/12345`, `https://contacts.l42.eu/people/alice`, etc. These should resolve to a user-facing page for the thing they identify, served by whichever service owns that UI.
- **Vocabulary URIs** identify ontology terms — `<…/ontology#trackLanguage>`, `<…/ontology#hasCategory>`, etc. These should resolve to the ontology document that defines the term, served by whichever service generates and exposes that document.

These are different conceptual namespaces. They may happen to coincide on the same hostname (when one service handles both jobs) or live on different hostnames (when responsibilities are split across services). The two cases differ only in deployment topology, not in principle.

`lucos_media_metadata_api` is the split case. There are two distinct services involved, despite their similar names:

- **`lucos_media_metadata_api`** (Go). This is the API service that ingests track data and serves the RDF export at `/v2/export` and the ontology document at `/ontology`. Its `APP_ORIGIN` is `https://media-api.l42.eu` in production.
- **`lucos_media_metadata_manager`** (PHP). This is the user-facing track-management UI. Its `APP_ORIGIN` is `https://media-metadata.l42.eu` in production. It hosts the per-track and per-collection pages users link to.

The Go API has a cross-service env var, `MEDIA_METADATA_MANAGER_ORIGIN`, that points at the PHP manager. That's the standard lucos pattern for referencing another service — the same shape as `LOGANNE_ENDPOINT` or `LUCOS_CONTACTS_URL`. The Go API uses this variable when constructing resource URIs, because the resource pages those URIs refer to live on the PHP manager.

The bug is that this same variable was also used when constructing vocabulary URIs — `mediaMetadataManagerOrigin + "/ontology#trackLanguage"`, etc. The vocab URIs therefore ended up at `media-metadata.l42.eu/ontology#…`, a hostname that doesn't serve the ontology document (the PHP manager has no `/ontology` route — it 404s). The document is served by the Go API at `media-api.l42.eu/ontology`. Vocab URIs and document hostname were silently divergent.

This caused two known issues:

- **`lucos_arachne#479`**: phase 2 of the `dcterms:language` → `mmm:trackLanguage` migration (PR `lucas42/lucos_arachne#476`) bound the arachne consumer's `MMM` namespace to `https://media-api.l42.eu/ontology#…`, matching the document hostname. Phase 1 of that migration in `lucas42/lucos_media_metadata_api#226` continued emitting at `media-metadata.l42.eu/ontology#…`, matching the existing producer convention. Consumer and producer disagreed on the URI; the Typesense `language` facet would have been wiped on the next `cleanup_searchindex` run. Stop-gapped via `lucas42/lucos_arachne#480` reverting the consumer to match the producer's (wrong-but-self-consistent) convention.
- **A separate latent bug** at `lucos_arachne/ingestor/searchindex.py:11`: `MEDIA_MANAGER_ONTOLOGY = Namespace("https://media-metadata.l42.eu/ontology/")` uses a trailing `/` rather than `#`. Used at line 209 to look up `mmm:onAlbum`, the resulting `URIRef` is `https://media-metadata.l42.eu/ontology/onAlbum` (slash form). The producer emits `https://media-metadata.l42.eu/ontology#onAlbum` (fragment form, `lucos_media_metadata_api/rdfgen/rdf.go:56`). rdflib doesn't bridge the two string forms, so `onAlbum` reads have been silently returning nothing on `main` since the constant was introduced.

Both bugs share a root cause: the producer's vocabulary URIs were never anchored to a hostname that anyone could expect to resolve. Once that anchoring is wrong, consumer code can drift in any direction and nobody notices until a query returns zero results.

## Decision

**Vocabulary URIs emitted by a service use that service's `APP_ORIGIN`.** No new env var is introduced. `APP_ORIGIN` is auto-provided by `lucos_creds` to every lucos service in every environment; it always points at the hostname where the service itself is served. Since the service that emits a vocabulary is the same service that serves the corresponding ontology document (the document is generated by the service from its own code), `APP_ORIGIN` is structurally the right pointer.

In `lucos_media_metadata_api` specifically: `APP_ORIGIN=https://media-api.l42.eu` in production, `http://localhost:3002` in dev. All `mmm:*` vocab URIs use this. Resource URIs (`/tracks/N`, `/collections/Slug`) continue to use `MEDIA_METADATA_MANAGER_ORIGIN` because they correctly point at the separate PHP manager.

### Cross-estate principle

> **Vocabulary URIs emitted by a service MUST use that service's `APP_ORIGIN`.** `lucos_creds` auto-provides `APP_ORIGIN` to every service; it always points at where the service is served, which is structurally where the ontology document lives. Use it.
>
> **Resource URIs emitted by a service use the `APP_ORIGIN` of whichever service hosts the user-facing resource page.** For a monolithic service that handles both jobs (the common case), that's the emitter's own `APP_ORIGIN` — same hostname for vocabulary and resources. For a split estate like `lucos_media_metadata_api` ↔ `lucos_media_metadata_manager`, the emitter references the other service's `APP_ORIGIN` via the standard cross-service env var pattern.
>
> **Rule of thumb for grep-discoverability:** if you find code constructing a `…/ontology#…` URI using anything other than `APP_ORIGIN`, you have taken a wrong turn. Either the URI is not actually a vocabulary URI and is misnamed, or the env var being used is the wrong one. Stop and reconsider before shipping.

Future services adding ontology terms inherit this principle by default. They should not need to think about it — using `APP_ORIGIN` is the path of least resistance and the right answer.

## Alternatives considered

### Introduce a new env var `MEDIA_API_ORIGIN` (rejected)

The first design proposal (in the team-lead reply that preceded this ADR) suggested introducing a new service-specific env var, `MEDIA_API_ORIGIN`, to point at the ontology document's hostname. The producer would then thread two distinct origin variables — `MEDIA_METADATA_MANAGER_ORIGIN` for resource URIs and `MEDIA_API_ORIGIN` for vocab URIs.

Rejected on lucas42's pushback. `APP_ORIGIN` already exists, is auto-provided in every environment, and is *exactly* the hostname we want for vocab URIs (since the service is what serves the ontology document). Inventing a service-specific variable for the same job is redundant — it adds estate-level configuration surface, requires production credentials work, and doesn't generalise to other emitters (each would need its own `<SERVICE>_ORIGIN` variable for the same role). `APP_ORIGIN` is the variable lucos already standardised on for "where this service is served"; using it for vocab URIs is the answer that scales.

### Dual-emit transition with `owl:sameAs` bridging (rejected)

The first design proposal also suggested a three-phase transition: Phase A (producer emits both old and new URIs in parallel, with `owl:sameAs` declarations bridging them), Phase B (consumer switches), Phase C (producer drops old). The dual-emit phase would have avoided any consumer/producer mismatch window.

Rejected by lucas42 as too much complexity for the small operational cost. The realistic window between a producer Phase 1 ship and a consumer Phase 2 ship is hours at most. The user-visible cost during that window is one ingest cycle of empty Typesense `language` facet, which recovers organically on the next ingest after Phase 2 ships. The engineering cost of dual-emit is doubled producer-side code, two extra deployments, and `owl:sameAs` semantics that interact awkwardly with the existing OWL inferencer. The trade did not justify the complexity.

The 2-phase plan adopted below explicitly accepts the brief mismatch window. The arachne ingestor does DROP+INSERT per source graph (ADR-0002), so old URIs are removed atomically on the first post-Phase-1 ingest — there is no protracted half-state in the triplestore.

### 301-redirect `media-metadata.l42.eu/ontology` → `media-api.l42.eu/ontology` (deferred)

Adding a 301 redirect on the PHP manager so that any old vocab URIs that may have been cached or bookmarked resolve to the canonical document.

Deferred. No identified concrete consumer of old URIs exists today: the internal slash-form `MEDIA_MANAGER_ONTOLOGY.onAlbum` lookup is fixed inside the Phase 2 ticket, removing the only known consumer that would otherwise be looking for the old URIs. External consumers do not exist (arachne is internal-only at this point). Adding a permanent redirect route to the PHP manager would be a small but lasting maintenance cost for no demonstrable benefit. Trivially addable later if a concrete consumer surfaces.

## Sequencing

**Two phases. No dual-emit. Brief consumer/producer mismatch window explicitly accepted.**

- **Phase 1** (`lucos_media_metadata_api`, `lucas42/lucos_media_metadata_api#227`). Producer adds `appOrigin := os.Getenv("APP_ORIGIN")` alongside the existing `mediaMetadataManagerOrigin`, and switches every `mediaMetadataManagerOrigin + "/ontology#…"` callsite to `appOrigin + "/ontology#…"`. `APP_ORIGIN` is already provided by `lucos_creds` in every environment, so no `lucos_creds` change is needed. Resource URIs are untouched.
- **Phase 2** (`lucos_arachne`, `#483`). Consumer switches the namespace constants in `ingestor/searchindex.py` from `media-metadata.l42.eu/ontology#…` to `media-api.l42.eu/ontology#…`, matching the new producer output. The latent slash/fragment bug at line 11 is fixed in the same change.

Between Phase 1 shipping and Phase 2 shipping — measured in hours at most — the consumer is reading the old URIs while the triplestore (after the first post-Phase-1 ingest) only has the new ones. The Typesense `language` facet will briefly empty during the affected ingest cycle. It recovers organically on the next ingest after Phase 2 ships.

`lucos_arachne#479` is stop-gapped via PR #480 (consumer reverted to match the existing producer convention). That stop-gap will be reversed by Phase 2 once Phase 1 is live — same line, same file, just the new hostname.

## Consequences

### Positive

- **Vocab URIs resolve.** Dereferencing `<https://media-api.l42.eu/ontology#trackLanguage>` lands on the actual ontology document, as it should. Future readers — human or agent — can follow a vocab URI to find out what the term means without guessing which other service might be serving the document.
- **The fix uses existing standard machinery.** `APP_ORIGIN` was already there. No new env var, no `lucos_creds` coordination, no per-service custom configuration. Future services adding ontology terms inherit the principle by default.
- **The cross-estate principle is documented in one place.** Other RDF emitters auditing themselves against ADR-0003 (tracked in `#484`) can check a single rule: do vocab URIs use `APP_ORIGIN`?
- **One latent bug fixed alongside.** The slash/fragment mismatch at `searchindex.py:11` has been silently returning empty `onAlbum` reads on `main` since the constant was introduced. Phase 2 fixes it as a side-effect of touching the same lines.
- **No data migration.** The arachne ingestor's DROP+INSERT-per-source pattern (ADR-0002) means the triplestore content reshapes atomically on the first ingest after Phase 1. No backfill script, no SPARQL UPDATE migration, no dual-state period.

### Negative

- **Brief Typesense facet wipe during cutover.** The `language` facet (4,956 tracks across ~30 languages) will be empty during the ingest cycle between Phase 1 ship and Phase 2 deploy. Users hitting the searchindex via seinn or arachne_explore will see no language filter for that window. lucas42 has explicitly accepted this cost as the right trade against dual-emit complexity.
- **The principle is enforced by convention, not by structure.** Nothing in the codebase prevents a future contributor from constructing a vocab URI using `mediaMetadataManagerOrigin` (or some other env var) again. The mitigation is the cross-estate audit (`#484`) and the grep-discoverable rule of thumb in this ADR.

### Neutral

- **Old URIs become unresolvable after Phase C deploy.** Anything in the wild still pointing at `https://media-metadata.l42.eu/ontology#…` will 404. No known consumers exist; if any surface, the deferred 301-redirect option above can be revisited cheaply.

## Implementation

The implementation work is tracked in three sibling issues filed alongside this ADR:

- **Phase 1 producer:** `lucas42/lucos_media_metadata_api#227` (blocked by this ADR).
- **Phase 2 consumer:** `lucos_arachne#483` (blocked by Phase 1 + one ingest cycle).
- **Cross-estate audit:** `lucos_arachne#484` (read-only audit of other RDF emitters; blocked by this ADR).

The audit is read-only — it identifies which other emitters (`lucos_contacts`, `lucos_eolas`, `lucos_photos`, `lucos_weightings`, `lucos_mood`, `lucos_schedule_tracker`) have the same conflation pattern. lucas42 decides separately whether to commission migration tickets off the back of the findings.
